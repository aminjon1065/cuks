# D2 — нормализация: типы ЧС, даты, уровень (severity)

Артефакты фазы **D2** (`docs/plan/DATA-INTEGRATION.md`). Готовят «грязные» поля исторических реестров
ЧС к вставке в `app.incidents`: вид ЧС (`type_code`), дата (`occurred_at`), уровень (`severity`).

## 1. `type-map.csv` — вид ЧС (ОФАТХО → incident_type.code)
Сопоставление **377 свободно-текстовых типов** (русский/таджикский, опечатки, составные) с кодами
справочника `incident_type`. Колонки: `raw_token, primary_code, secondary_codes(pipe), verified, freq, note`.

- **Покрытие:** 377/377, **0 невалидных кодов**, 0 UNKNOWN. 284 подтверждены состязательной проверкой;
  остальные — из первичного маппинга (keyword-sanity-чек на 90 непроверенных дал **0 противоречий**).
- **Составные события** (168 токенов): `primary_code` = доминирующий вид, прочие → `secondary_codes`.
  Напр. `дождь, град, сель → nat.geo.mudflow + nat.meteo.downpour|nat.meteo.hail`.
- **Топ:** сель 1238 · утопление 543 · землетрясение 477 · лавина 450 · наводнение 411 · ливень 396 записей.
- **Справочник расширён** (решение заказчика — всехазардный реестр): добавлены `accident` (Происшествие) →
  `accident.drowning`/`electrocution`/`other` и `nat.meteo.lightning`. ДТП/пожар → существующие `tech.*`.
- `_review-types.csv` — 5 неоднозначных токенов (прорыв, СБ, трансграничный конфликт…) для ручной сверки.

> Построен многоагентным workflow (маппинг + состязательная проверка). Прогон дважды прерывался
> транзиентными сбоями API субагентов — итог **восстановлен из кэша journal** (9/9 map-батчей, 7/9 verify)
> и детерминированно доведён (1 добор + 4 override + валидация). Достоверность не пострадала.

## 2. Правило даты (`occurred_at`) — заземлено на данных
Полнота в источниках: **год ~99%, месяц ~99%, день ~95%** (в 1992-2016 — 748 диапазонов вида «25–26»).
Правило: `occurred_at` = `YEAR`-`MONTH`-`DAY` в 12:00 Asia/Dushanbe → UTC.
- День-диапазон («25–26») → берём **первый** день.
- Нет дня → **15-е**; нет месяца (редко) → **1 июля**; нет года → строка во «flag», не вставляется.
- `reported_at` = `occurred_at` (отдельного времени донесения в данных нет; инвариант `reported_at ≥ occurred_at`).

## 3. Правило уровня (`severity` 1–5) — производное
Явного уровня в данных **нет** ни в одном файле → выводим эвристикой из доступных сигналов (все разрежены):
`dead` (структурно только 2018–2020), `injured`, `evacuated`, `affected`, `damage_est` (сумма ~9%).
- `dead ≥ 5` **или** `damage_est ≥ 1 млн сомони` → **4** (региональный);
- `dead ≥ 1` **или** `affected ≥ 100` **или** `damage_est ≥ 100 тыс.` → **3** (местный);
- иначе → **2** (объектовый) по умолчанию.
Уровень **грубый** (сигналы редкие); уточняется при парсинге текста жертв/ущерба в D3/D6. Правило фиксируется,
чтобы `severity` был воспроизводим и объясним, а не выдуман построчно.

## Использование (D3 ETL)
`type-map.csv` join по `raw_token` → `primary_code` (+ `secondary_codes` в `attrs`). Перед вставкой инцидентов
**пересеять справочник** (`pnpm --filter @cuks/db seed` — новые коды `accident.*`/`nat.meteo.lightning`
идемпотентно добавятся), иначе FK `incidents.type_code` не на что сослаться.
