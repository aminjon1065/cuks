## Повседневное администрирование (веб-админка)

Все действия — в разделе «Администрирование» (видим при наличии прав `admin.*`). Значимые операции пишутся в
аудит (`admin.audit.view`).

### Пользователи (`admin.users.manage`)
- **Создать**: Пользователи → «Добавить». ФИО, подразделение/должность, роли. Логин и временный пароль
  генерируются; пароль показывается ОДИН раз — передать сотруднику. При первом входе — форс-смена пароля и
  (для привилегированных ролей) обязательная привязка 2FA.
- **Импорт списком**: мастер CSV для первичного наполнения (валидация перед загрузкой).
- **Карточка**: сброс пароля (новый временный), **сброс 2FA** (при потере устройства), блокировка/
  разблокировка (блокировка рвёт активные сессии < 5 c), отзыв сессий, сертификаты ЭЦП, квоты файлов, журнал
  действий пользователя.
- **Увольнение**: блокировка + мастер передачи (открытые маршруты/задачи/владение → преемнику).

### Орг-структура (`admin.org.manage`)
Дерево подразделений и должностей (drag-and-drop), назначение руководителя. Перемещение сотрудника обновляет
каналы подразделений и скоупы прав.

### Роли и права (`admin.roles.manage`)
Матрица прав по модулям. Системные роли — только чтение (копировать как основу для своей). Назначение роли —
из карточки пользователя (роль + опциональный скоуп-подразделение). Изменения применяются без перелогина
(инвалидация ability-кэша).

### Справочники (`admin.dicts.manage`)
Виды ЧС, типы документов, корреспонденты, номенклатура дел, шаблоны маршрутов, метки. Деактивация вместо
удаления, если код используется.

### Аудит (`admin.audit.view`)
Журнал всех значимых событий с фильтрами (период, актор, действие, сущность, IP) и выгрузкой в CSV. Отдельная
вкладка — журнал чтения ДСП.

### Здоровье платформы (`admin.system.monitor`)
Статусы сервисов, размеры БД/бакета/диска, очереди BullMQ (с «Повторить» для упавших), последний бэкап, ошибки
за 24 ч. Подробнее — runbook-monitoring.md.

## Демо-режим (показ руководству)

Для презентации на **непрод**-стенде есть обучающие сиды, наполняющие платформу связным набором данных:
демо-пользователи (разные роли — видно разницу прав), кластер ЧС по стране с хронологией, карточки на
доске «Оперативные поручения» и оперативная переписка в канале. Запуск:

```bash
pnpm db:seed:demo          # = seed + демо-контент; идемпотентно (повторный запуск не дублирует)
```

- Демо-пользователи входят с паролем `Demo!2026` (переопределяется `SEED_DEMO_PASSWORD`). Сравните
  `nazarova.n` (сотрудник) и `yusupov.f` (администратор платформы) — разный UI по правам.
- Отказоустойчиво: `--demo` **запрещён** при `NODE_ENV=production` (у демо-аккаунтов известный пароль).
- Демо-данные помечены (`ЧС-DEMO-*`, фиксированные id карточек) и накладываются поверх базовых сидов, не
  затрагивая реальные записи.

## Регулярные операции

- **Ежедневно**: беглый взгляд на health-дашборд (сервисы зелёные, очереди без роста `failed`, был ночной
  бэкап).
- **Еженедельно**: проверить off-site-копию бэкапа; свободное место на диске.
- **Ежеквартально**: restore-drill (runbook-backup.md), пред-прод security-чеклист при обновлениях
  (runbook-security.md), аудит `admins-without-2fa.sql`.
- **При обновлении**: миграции ДО перезапуска, окно обслуживания вечером, бэкап перед началом.
