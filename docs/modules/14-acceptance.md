# Модуль 14 — Приёмка звонков/записей (§9): ручной прогон

Автотесты закрывают детерминируемую часть §9 (права на запись — `apps/web/e2e/meet-acceptance.spec.ts`;
ring/decline/missed — `meet-ring.spec.ts`; комнаты/токен/host — `meet-rooms.spec.ts`; встречи —
`meet-meetings.spec.ts`; смоук пре-джойна — `meet-ui.spec.ts`). Критерии, требующие реальной
WebRTC-медиасессии (несколько браузеров с камерой/микрофоном, egress, TURN), проверяются вручную по
этому чеклисту на стенде.

## Предусловия стенда

- Реальный домен + TLS (WebRTC/WebCrypto/Secure-cookie требуют https) — либо внутренний CA Caddy
  (`docs/08-deployment.md`).
- LiveKit + **Egress** подняты: в dev — `docker compose -f infra/docker/compose.dev.yaml --profile recording up -d`
  (Egress вне профиля по умолчанию не стартует). После migrate/seed — `docker restart` при необходимости.
- `LIVEKIT_URL/API_KEY/API_SECRET` заданы в `.env` и совпадают с `infra/docker/livekit/livekit.yaml`.
- Прод: TURN `external_tls` на `5349/tcp` за Caddy (задача 7.1); порты 7881/tcp, 7882/udp (dev) /
  50000-50200/udp (prod) открыты (`docs/08-deployment.md` §firewall).
- 2–8 участников на разных устройствах/вкладках с камерой и микрофоном.

## Чеклист

### 1. Пре-джойн (§9)

- [ ] Открыть комнату по ссылке `/app/meet/r/{slug}`.
- [ ] **Без камеры**: выбрать «выключить камеру» на пре-джойне → войти → аудио двустороннее, видео нет.
- [ ] **Зритель (без обоих устройств)**: выключить камеру и микрофон → войти → видно/слышно других,
      сам не публикует.
- [ ] Тест микрофона показывает уровень; выбор камеры/микрофона/динамиков работает.

### 2. Звонок 1:1 из DM (§9)

- [ ] Из DM нажать 📹 → у получателя всплывает входящий (мелодия) → **принять** → двусторонние
      видео/аудио.
- [ ] Завершить звонок → в DM системная карточка звонка; длительность отображается (см. примечание ниже).
- [ ] Повторить с **отклонить** → карточка «Звонок отклонён».
- [ ] Повторить и **не отвечать 30 с** → карточка «Пропущенный звонок».

### 3. Звонок в канале + баннер (§2)

- [ ] Из канала нажать 📹 → в канале у остальных участников появляется баннер «Идёт звонок
      [Присоединиться]».
- [ ] Присоединиться по баннеру → попадание в ту же комнату.
- [ ] После ухода всех — баннер гаснет (webhook `room_finished`).

### 4. Конференция 8+ участников (§9)

- [ ] 8 участников в одной комнате; грид/спикер-раскладка, пагинация при >25.
- [ ] Двое одновременно включают **демонстрацию экрана** (приоритет screen-share в раскладке).
- [ ] Стабильность **≥ 30 минут** без деградации/утечек.
- [ ] У одного участника **отключить сеть на 10 с** → авто-reconnect, восстановление медиа.
- [ ] Индикаторы качества соединения и «говорит» (рамка) корректны; реакции/поднятие руки/чат комнаты
      работают; push-to-talk (пробел) не оставляет «горячий» микрофон при потере фокуса.

### 5. Host-действия (§3)

- [ ] Ведущий: выключить микрофон участнику; «выключить всем»; удалить участника; завершить для всех.
- [ ] Участник не имеет host-контролов (роль из `myRole`, сервер перепроверяет `created_by`).

### 6. Запись (§4/§9)

- [ ] Ведущий с правом `meet.record` нажимает «Начать запись» → **индикатор записи виден ВСЕМ** +
      системное объявление о старте.
- [ ] Остановить запись → объявление об остановке.
- [ ] Файл появляется на странице «Записи» (статус processing → ready после egress-webhook), **играет в
      браузере** (стриминг с range).
- [ ] **Доступ только участникам** — покрыто автотестом (`meet-acceptance.spec.ts`): не-участник получает
      403 на карточку/stream/download, участник и `meet.recordings.manage` — доступ.
- [ ] Скачивание пишет событие в `audit_log` (`meet.recording_downloaded`).
- [ ] Удаление (ведущий/manage) убирает запись и файл.
- [ ] Одновременных записей **не более 2** (третья — сообщение «Запись начнётся после освобождения
      слота»); повторный старт на одну комнату не создаёт вторую запись.

### 7. Работа через TURN (§9)

- [ ] Клиент за «жёстким» NAT — эмулировать блокировкой исходящего **UDP** (диапазон медиа + 7882/udp).
- [ ] Соединение устанавливается через встроенный TURN LiveKit по **`tls:5349`** (проверить в
      chrome://webrtc-internals: candidate типа `relay`).

## Примечания / известные ограничения (перенесено из STATUS)

- **Длительность звонка в системной карточке** и авто-карточки «начался/завершён» — требуют трекинга
  конца звонка через webhook (meet_calls); в 6.4 реализованы missed/declined, длительность — follow-up.
- **Предупреждение за 14 дней** до удаления записи — нужен API-диспетчер outbox (follow-up 6.6).
- **Полноценная очередь** «запись начнётся после освобождения слота» — сейчас 409 с сообщением.
- **dev-ICE для медиа** — на Docker Desktop через bridge-сеть может понадобиться `rtc.node_ip`/
  `use_external_ip` для реального браузера (см. STATUS 6.1).
