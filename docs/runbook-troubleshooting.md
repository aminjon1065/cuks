# Типовые сбои — runbook администратора

Диагностика и устранение частых проблем платформы CUKS (задача 7.7). Первый шаг почти всегда — открыть
админский health-дашборд (`/app/admin/health`, право `admin.system.monitor`) и посмотреть, какой сервис
«красный». Команды даны с алиасом из docs/runbook-install.md:

```bash
alias dc='docker compose --env-file .env -f infra/docker/compose.prod.yaml'
```

## Быстрая диагностика

```bash
dc ps                              # статусы + health контейнеров
dc logs -f --tail=200 api          # логи api (аналогично worker, caddy, livekit…)
curl -fsS https://<домен>/api/health/ready   # up/down по каждой зависимости
dc stats --no-stream               # CPU/RAM по контейнерам
df -h                              # место на диске (частая причина каскадных сбоев)
```

## Каталог проблем

### На карте нет слоёв / тайлы не грузятся
**Симптом**: карта открывается, но подложка/слои пустые.
**Причина**: Martin кэширует каталог таблиц при старте. После `db:migrate`/`db:seed` (появились новые
таблицы-слои) он о них не знает.
**Решение**: `dc restart martin`. Проверить `dc logs martin` — должен перечислить источники. Если базовая
подложка отсутствует — проверить, что `infra/basemap/region.pmtiles` собран (`infra/scripts/build-basemap.sh`).

### Не входит в систему
- **«Неверный логин или пароль»** при верных данных → аккаунт мог быть заблокирован (админка → Пользователи →
  разблокировать) или сработал lockout (5 неудач за 15 мин, docs/05 §1) — подождать окно или сбросить.
- **Требует 2FA, а устройство потеряно** → админ: карточка пользователя → «Сбросить 2FA»; при следующем входе
  пользователь заново привяжет аутентификатор.
- **Циклический редирект на смену пароля** → у пользователя `must_change_password`; это нормально при первом
  входе, он должен задать новый пароль.
- **Сессии сбрасываются сразу после входа** → проверить, что за Caddy отдаётся HTTPS и cookie `Secure`
  проходит (WebCrypto/Secure-cookie требуют TLS); при `tls internal` — установлен ли корневой CA на рабочей
  станции (docs/runbook-install.md §7).

### Файл не загружается / «на проверке» навсегда
- **Загрузка обрывается** → размер больше лимита (2 ГиБ) или MinIO недоступен (`/admin/health` → minio).
- **Файл вечно «на антивирусной проверке»** → ClamAV не поднялся или ещё грузит базы (первый старт ~1–2 мин):
  `dc logs clamav`, `dc ps` (health), при необходимости `dc restart clamav`. Воркер не может просканировать —
  файл остаётся `pending`.
- **Скачивание presigned-ссылки не работает** → S3_PUBLIC_ENDPOINT/поддомен `s3.<домен>` и его TLS; проверить
  `dc logs caddy` на ошибки маршрута.

### Звонок/конференция не соединяется
- **Собеседники не видят/не слышат друг друга** → медиа-порты. Открыт ли UDP 7882 (основной путь) и TCP 7881
  на файрволе? За строгим NAT — включить встроенный TURN (docs/runbook-install.md §8).
- **Запись не появляется** → egress-контейнер: `dc logs egress`; проверить, что `LIVEKIT_API_SECRET` в egress
  и livekit совпадают (иначе вебхуки/egress отклоняются).
- **Баннер звонка «висит»** после завершения → это чинит вебхук `room_finished`; если LiveKit не достучался до
  `api:3000/api/meet/webhook` — проверить сеть/логи api.

### Уведомления/почта не приходят
- Проверить очередь `email` на health-дашборде (waiting/failed) и `SMTP_URL`. Тест — админка → Настройки →
  «Отправить тестовое письмо». Офлайн-пользователю уведомление дублируется в email (docs/16 §B).

### Очередь задач растёт / задачи «failed»
- Health-дашборд → Очереди: если `failed > 0` — кнопка «Повторить»; если растёт `waiting` при нулевом
  `active` — воркер не обрабатывает: `dc ps worker`, `dc logs worker`, `dc restart worker`.

### Сервис «красный»/перезапускается
```bash
dc ps                         # найти unhealthy/restarting
dc logs --tail=200 <service>  # причина в логах
dc restart <service>          # мягкий рестарт
```
Частые причины: нехватка памяти (см. `mem_limit`, `dc stats`), заполненный диск, неверная переменная в `.env`
(api/worker падают на валидации env при старте — сообщение в логах перечисляет отсутствующие переменные).

### Диск заполнен
- Виновники обычно: записи конференций (бакет MinIO), логи Docker (ограничены `json-file 50m×5`), бэкапы
  (`backupdata`). Смотреть `df -h`, размеры на health-дашборде. Retention записей чистит воркер (docs/14 §4);
  бэкапы ротируются `--keep-daily 30 --keep-monthly 12`. Уносить старые бэкапы off-site (docs/runbook-backup.md).

### После обновления что-то сломалось
- Миграции применяются ДО перезапуска (`dc run --rm api pnpm db:migrate`). Если api не стартует — смотреть,
  применились ли миграции и совпадает ли схема. Откат — восстановление из бэкапа (docs/runbook-backup.md),
  миграции назад не откатываются автоматически.

## Эскалация

Если проблема не локализуется: собрать `dc logs --since=1h` затронутых сервисов, вывод `dc ps`, `df -h`,
скриншот health-дашборда и передать разработчику. Данные не терять — при сомнениях сначала снять бэкап
(`dc exec backup backup.sh`), потом чинить.
