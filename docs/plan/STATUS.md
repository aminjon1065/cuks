каждый finding опровергается двумя скептиками — линзы reachability + mechanism; **4 находки подтверждены

осознанное поведение вне спеки):

- **[MED] `= ANY(co_executors)` не использует GIN-индекс** (`myTasksDocumentIds`/`mine`): при росте
  таблицы `my_tasks` уходил в seqscan мимо `resolutions_co_executors_idx`. **Исправлено**: `arrayContains`

- **[MED] `create()` отклонял только draft/on_route**: резолюцию можно было выдать на
  `pending_registration` и терминальные (`completed`/`archived`/…). **Исправлено**: только

- **[LOW] дата-срок показывалась как `05:00`**: `<input type="date">` → полночь-UTC → +5ч в Душанбе.

- Отмена родителя **не каскадит** на детей (ревью не сочло дефектом — спека §5 молчит; дети — отдельные
  инструкции с собственным автором). Каскад/эскалации — в контроле 3.8.
- Срок хранится как timestamptz (полночь-UTC для дат без времени); отображается как дата. Точное время
  дедлайна (конец дня) — вопрос контроля 3.8, не 3.4.

**Известные проблемы**:

### 2026-07-15 — Фаза 3: задача 3.3 (маршруты: движок sequential/parallel, reject-цикл, шаблоны)

- **`feat(docflow)`**: чистый **движок активации** (`route-engine.ts`) — группы по `step_order`
  активируются по порядку, группа завершена → активируется следующая, последняя → маршрут завершён.
  `RoutesService`: **отправка по маршруту** (из шаблона или ручные шаги) draft→on_route;
  **согласование** двигает группы, завершение → doc `pending_registration`; **отклонение** (с причиной)
  отменяет маршрут, doc→draft. Согласование конкурентно сериализуется блокировкой строки маршрута
  (`FOR UPDATE`) + перечитыванием шага. Разрешение исполнителя: сам / должность / глава подразделения;
  на этом же — очередь «На согласование» и **видимость участникам маршрута** (расширяет базовую 3.2,
  вынесенную в общий `document-visibility`). Ручные переходы `on_route`/`pending_registration` убраны из
  графа статусов (движок владеет ими напрямую). Endpoints approve/reject + CRUD шаблонов.
- **`feat(web)`**: карточка → секция **«Маршрут»** — степпер по циклам (шаги: исполнитель, статус,
  решение, комментарий, время), кнопки согласовать/отклонить на активном шаге пользователя, диалог
  «Отправить по маршруту» (серверный поиск согласующих → последовательные шаги). Кабинет → вкладка
  **«На согласование»**. Namespace `docflow` расширен (ru + tg-заглушка, паритет).

**Тесты/проверка**:

- Корневой gate typecheck/lint/test/build — зелёный: **API 257** (+route-engine, статус-машина),
  **web 152** (+RouteSection/StartRouteDialog), shared/worker/ui/db. Новые unit: `planApproval`/
  `stepsToActivateOnStart` (движок), web компоненты маршрута.
- **Живая проверка** (реальный API + Postgres 17, e2e Playwright, 7 docflow-тестов зелёные):
  двухшаговый маршрут — согласующий 1 двигает → согласующий 2 завершает → doc `pending_registration`;
  **отклонение** возвращает doc в `draft` (причина обязательна, 400 без неё); плюс регресс-тесты 3.1/3.2.

**Adversarial-review** (4 измерения: authz/canAct/видимость, движок+конкурентность, sql/схема, веб →
каждый finding опровергается двумя скептиками; **1 находка → подтверждена двумя голосами, исправлена**;
authz/движок/sql — чисто):

- **[HIGH] пикер согласующих ограничен 20 пользователями**: диалог грузил каталог пустым запросом
  (`useDirectoryUsers('')`), а `UserPicker` фильтрует клиентски — согласующий за пределами первых 20
  (кап каталога) недобираем. **Исправлено**: серверный поиск (инпут → `useDirectoryUsers(search)` +
  список результатов, как в files `ShareDialog`); любой пользователь достижим. Регресс-тест.

**Решения**:

- Маршрут владеет переходами статуса документа (draft→on_route, on_route→pending_registration,
  on_route→draft) — ставит их напрямую (как register()); ручной `changeStatus` в эти состояния не ведёт.
- Видимость участникам маршрута: назначенный согласующий видит документ (нельзя согласовать то, что не
  видишь). Для ДСП это осознанный грант через назначение (ревью не сочло утечкой). Участники резолюций и
  руководство подразделения-владельца — 3.4/3.8.
- 3.3 действует по шагам `approve`; `sign`→3.5, `acknowledge`→3.6. «Отправить по маршруту» — ручные
  последовательные шаги (шаблоны — альтернативный путь; CRUD шаблонов заведён под канцелярией).

**Известные проблемы**:

- e2e `docflow-documents-ui` (создание+регистрация через UI) флапает при холодном старте под 7
  параллельными воркерами: кнопка «Создать документ» ждёт права из `/auth/me`, которые на холодном
  старте не успевают. В одиночку/на тёплых серверах — зелёный. Робастность харнесса, не дефект кода.
- Инфра: рассинхрон redis compose(`6379`)/`.env`(`6380`) — e2e гонялся с override (см. запись 3.2).
- Отклонение отменяет весь маршрут; параллельные шаги той же группы остаются `active` под отменённым
  маршрутом (действовать нельзя — проверка `route.status='active'`), в `skipped` не переводятся —
  косметика.

подтверждены, все LOW):

- **Деньги — переполнение**: `sum(damage_est)::numeric(18,2)` мог переполниться на гигантских
  суммах и уронить эндпоинт (500). **Исправлено** на `round(…,2)::text` (и в stats, и в kpis 2.10).
- **Хороплет — регионы без данных**: при фильтре регионы без ЧС рисовались светлым дефолтом ECharts
  (ломало тёмную тему). **Исправлено**: базовый `areaColor: theme.surface`.
- **Хороплет — лишние перерисовки**: объект `map` не мемоизировался → карта перестраивалась на
  каждый ре-рендер. **Исправлено** `useMemo`.
- **Принято** (опровергнуто как недостижимое): `sum(dead/injured/evacuated)::int` теоретически
  переполняется при суммарных жертвах >2^31 — физически невозможно (≈200× населения Таджикистана);
  `::int` сохраняет тип `number` (bigint вернулся бы строкой). См. «Известные проблемы».

**Решения**:

- Хороплет и «топ районов» — на уровне **регионов**: в dev нет геометрии районов (все 188 ЧС
  привязаны к региону, ни один — к району). Запросы группируют по `region_id`; при импорте границ
  районов (`seed-geo.sh` ADM2) те же запросы дают районы. — выбор заказчика.
- Границы для ECharts отдаёт новый `/analytics/regions.geojson` — в приложении не было GeoJSON-
  выдачи `admin_units` (только MVT-тайлы Martin).
- ECharts **5** (не 6, по docs/02); lazy-загрузка, чтобы ~1 МБ echarts не попадали в начальный
  бандл главной.
- Временные корзины считаются в Asia/Dushanbe; tz — SQL-литерал (`sql.raw`), иначе drizzle сделал бы
  bind-параметр и SELECT≠GROUP BY.
- Период — скользящее окно 30д/90д/год (год по умолчанию), явные from/to.

**Известные проблемы**:

- Районный уровень хороплета/топа активируется после импорта границ районов (ADM2, `seed-geo.sh`).
- `sum(жертвы)::int` переполнится при суммарных жертвах >2^31 (недостижимо); чистый фикс потребовал
  бы bigint→строку и сломал бы тип `number`. Принято осознанно.

**Решения**:

- **`gdal-async` (GDAL 3.12, prebuilt, worker-only, ~215 МБ) вместо системного `ogr2ogr`** — тот же
  движок, но программный API даёт построчный лог ошибок (§6) и не тащит креды БД в argv дочернего
  процесса. Зафиксировано как одобренная тяжёлая зависимость (docs/02).
- **Цель репроекции — `fromProj4('+proj=longlat +datum=WGS84 +no_defs')`, а не `fromEPSG(4326)`**:
  GDAL 3 уважает авторитетный порядок осей EPSG:4326 (широта-первой) и молча свапает координаты
  (проверено: 3857→EPSG:4326 даёт 38.5, 68.8). proj4-форма не несёт авторитетного порядка осей.
- **Физическая таблица импортированного слоя при мягком удалении слоя НЕ дропается**: строка слоя
  soft-deleted (CLAUDE.md §2), тайлы перестают её отдавать (функция фильтрует `deleted_at`), а
  DROP — решение retention/2.9 (её может читать QGIS напрямую, §7). Табличный источник
  `layer_features` и физические `l_<slug>` остаются в авто-каталоге Martin для QGIS-клиентов (2.9).
- **Даты из источника импортируются как text**: форматы дат в источниках несогласованы, неудачный
  каст потерял бы всю строку.
- **Права по docs/05**: импорт — `gis.import`, экспорт — `gis.export`. Импорт/экспорт-записи
  per-user (не-суперадмин видит только свои — файл/выборка могут нести чувствительные данные).

**Известные проблемы**:

- Nest-сборки (`api`/`worker`) резолвят модули по node10 и игнорируют `exports`-карту пакета,
  поэтому глубокий импорт `@cuks/shared/office/xlsx` требует явного `paths` в `tsconfig.build.json`

- **Не исправлено (сознательно, задокументировано в коде)**: единый глобальный `pg_advisory_xact_lock`

- **`current_version_id` без FK на `file_versions`** — иначе цикл `fs_nodes ↔ file_versions`

- **Проверка дубликата имени — на уровне сервиса** (`SELECT` затем `INSERT`), не БД-constraint —
  малое окно гонки принято осознанно (упомянуто и во время ревью, не эскалировано выше low).
- **Размер папки не считается рекурсивно** — `size_cached` осмыслен только для `kind=file`;

- **`checksum_sha256` считается на клиенте** и отправляется на `/complete` — сервер никогда не видит
  сырые байты (presigned upload напрямую в MinIO), посчитать хэш на сервере значило бы скачивать файл
  обратно, убивая смысл прямой загрузки.
- **Публичные `/files/*`-эндпоинты 1.2 не включают ACL-шаринг** (`GET/PUT /files/:id/acl`, `POST
  /files/:id/links`, «Доступные мне») — задача 1.4; `assertAccess` уже готов принять точечные гранты

- **[count]** `employeeCount` теперь считает **уникальных активных** сотрудников

  детерминирован (orderBy).
- **[integrity]** `head_position_id` при update — только живая должность; удаление подразделения
  блокируется при живых scoped-назначениях ролей (`has_role_scopes`); аудит `primary_set`

- **ACL-хелперы** (`AclService`, level-3): grant/revoke/list + `check(user, resource, minLevel)`

- **Скоупы** (`ScopeService`, level-2): `getAccessibleOrgUnits(user, permission)` — глобально или
  список подразделений с раскрытием поддерева через materialized `path` (для фильтров в модулях).
- **Контроллер** `/api/v1/admin/{permissions,roles,role-assignments}` — все под
  `@RequirePermission('admin.roles.manage')`. Каталог permissions отдаётся с группировкой по
  модулям (RU-тексты — на фронте через i18n).
- Матрица прав применяется **без перелогина** (permissions читаются на каждый запрос — решение 0.4).

**Тесты/проверка**: `typecheck/lint/format/test/build` — зелёные (37 тестов; +rbac-хелперы,
createRoleSchema). **Live**: непривилегированный `operator` (роль employee) → `/admin/roles` 403
`permission.denied`; админ (после enroll 2FA) — каталог 35, роли 7, создание роли, PATCH системной
роли → 403 `system_readonly`, назначение со скоупом → 201, отзыв/удаление → 200.

**Отложено**: ACL/scope применяются в сервисах модулей по мере их появления (docs/05 §3 «правила —
в спеках модулей»).

**Ревью (adversarial).** Прогнан ревью RBAC; 8 находок исправлены (критичных нет):

- **[esc]** privilege-bounded delegation: держатель `admin.roles.manage` (не суперадмин) больше не
  может выдать себе роль superadmin (wildcard) — 403 `superadmin_forbidden`; и не может создать/
  назначить роль с правами, которых сам не имеет — 403 `permission_exceeds_grant`/`exceeds_grant`.
  Отзыв superadmin-назначения — только суперадмином. Проверено live.
- **[corr]** `create()` проверяет занятость кода только среди живых ролей (isNull deletedAt) —
  соответствует partial-unique; замена прав роли — в транзакции (нет окна «ноль прав»); `getOne`
  без re-list всей таблицы.
- **[acl]** фильтр soft-deleted позиций/подразделений в resolveUserSubjects/expandSubtrees/assign;
  дедуп результата поддерева; аудит `acl.revoked` с ресурсом/субъектом/уровнем.

**Документированные решения ревью**:
- Списки `/admin/{roles,permissions,role-assignments}` возвращают массивы (ограниченные
  конфиг-списки, не большие таблицы) — без пагинации docs/04.
- Level-3 `org_unit`-субъект ACL сопоставляется с **прямыми** подразделениями пользователя (без
  каскада по поддереву); level-2 scope, наоборот, раскрывает поддерево — разные семантики намеренно.
- [P2] Юнит/интеграционные тесты `AclService`/`ScopeService` требуют тест-БД (testcontainers/compose,
  docs/04) — вводятся с харнессом; сейчас покрыто live-smoke + shared-тестами хелперов.

  потребителем, чтобы не фиксировать форму без валидации.
- 2026-07-12 — **UUIDv7 генерируется на клиенте** (`uuidv7`, ADR-14) через `$defaultFn`, без
  PG-функции (в PG17 нет `uuidv7()`). Орг-скелет в сидах — с фиксированными UUID для
  идемпотентности и детерминированного `path`.
