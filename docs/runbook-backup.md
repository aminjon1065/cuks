# Бэкапы и восстановление — runbook администратора

Резервное копирование и восстановление платформы CUKS (docs/08-deployment.md §Бэкапы, docs/09 §6, задача
7.2). Цели: **RPO 24 ч** (ежедневный бэкап), **RTO 4 ч** (проверенное восстановление). Бэкап без
проверенного восстановления бэкапом не считается — см. §Restore-drill.

Все команды — из корня репозитория с алиасом из docs/runbook-install.md:

```bash
alias dc='docker compose --env-file .env -f infra/docker/compose.prod.yaml'
```

## Что и куда копируется

Сервис `backup` (образ из `infra/docker/Dockerfile` таргет `backup`) каждую ночь в 02:00 (Asia/Dushanbe):

| Данные | Метод | Замечание |
|---|---|---|
| PostgreSQL | `pg_dump -Fc` (логический, консистентный) | вся БД `cuks`; восстановление — `pg_restore` |
| MinIO (файлы, документы, записи) | снимок тома `miniodata` в restic | point-in-time |
| GeoServer конфиг | том `geoserverdata` | — |
| Внутренний УЦ (ключ ЭЦП) | том `ca_data` | **критично** — без него не проверить/выпустить подписи |
| Caddy (сертификаты) | том `caddy_data` | переиздаётся автоматически, но экономит ACME-лимиты |

Redis (сессии, очереди BullMQ) **не** копируется намеренно: это транзиентное состояние (потеря = повторный
вход пользователей и переустановка отложенных задач, в пределах RPO допустимо).

Репозиторий — **restic**, зашифрованный (`RESTIC_PASSWORD`), на томе `backupdata`. Ротация:
`--keep-daily 30 --keep-monthly 12`, затем `prune`. После каждого бэкапа — быстрый `restic check` (структура
+ 5% данных).

> **`RESTIC_PASSWORD` храните ОФФЛАЙН** (сейф админа / менеджер паролей). Потеря пароля = все бэкапы
> невосстановимы. Пароль не лежит в git; в `.env` (chmod 600) он есть, но `.env` — не резервная копия пароля.

## Ежедневная эксплуатация

```bash
# Список снапшотов (id, дата, теги)
dc exec backup restic snapshots

# Разовый бэкап вне расписания (перед обновлением/миграцией)
dc exec backup backup.sh

# Проверить целостность репозитория целиком (раз в месяц; читает все данные — долго)
dc exec backup restic check --read-data

# Использование/статистика репозитория
dc exec backup restic stats
```

## Хранение вне сервера (обязательно)

Локальный том `backupdata` защищает от потери БД/тома, но не от потери сервера. Скопируйте репозиторий на
внешний диск или второй хост (ротация — 30 дней + 12 месячных, минимум еженедельно):

```bash
# Вариант А — синхронизировать сам том restic на внешний диск/rsync-хост (репозиторий уже зашифрован):
docker run --rm -v cuks_backupdata:/repo:ro -v /mnt/external/cuks-restic:/dst alpine \
  sh -c 'apk add --no-cache rsync >/dev/null && rsync -a --delete /repo/ /dst/'

# Вариант Б — restic copy во второй репозиторий (SFTP/USB); пароль второго репо — тот же RESTIC_PASSWORD:
dc exec backup sh -c 'restic copy --repo2 sftp:backup@nas:/cuks-restic --password-file2 /run/secrets/restic'
```

Внешнюю копию тоже проверяйте restore-drill'ом (см. ниже) не реже раза в квартал.

## Restore-drill (проверка восстановления на чистой VM)

Раз в квартал. Цель — доказать, что из репозитория поднимается рабочая система за RTO. Выполнять на
отдельной чистой VM (не на проде), где есть Docker и копия репозитория restic.

1. **Код и .env.** Склонировать репозиторий, `cp .env.prod.example .env`, задать как минимум
   `POSTGRES_*`, `RESTIC_PASSWORD` (тот же, что шифровал бэкап), `RESTIC_REPOSITORY`-совместимость (том).
   Разместить восстанавливаемый restic-репозиторий в томе `cuks_backupdata` (скопировать внешнюю копию туда).
2. **Только данные-сервисы, пустые тома.**
   ```bash
   dc up -d postgres minio
   dc ps            # дождаться healthy
   ```
3. **Восстановление** одноразовым запуском `backup`-образа с томами, смонтированными на запись (в отличие от
   планового сервиса, где они `:ro`):
   ```bash
   dc run --rm --no-deps \
     -v cuks_miniodata:/data/minio -v cuks_geoserverdata:/data/geoserver \
     -v cuks_ca_data:/data/ca -v cuks_caddy_data:/data/caddy \
     backup restore.sh latest
   ```
   Скрипт: `restic restore latest` → `pg_restore --clean --if-exists` в БД → копирование объектов/конфигов в
   тома. (Без RW-монтирования тома пропускаются с явным предупреждением — только БД восстановится.)
4. **Поднять стек и мигрировать при необходимости.**
   ```bash
   dc run --rm api pnpm db:migrate    # обычно no-op: дамп уже на актуальной схеме
   dc up -d
   dc restart martin
   ```
5. **Проверка (smoke)** — по чеклисту docs/runbook-install.md §10: `/api/health/ready` зелёный; вход
   администратора; на карте есть слои; ранее загруженный файл скачивается; запись воспроизводится.
6. **Зафиксировать** дату, версию, время до готовности (должно укладываться в RTO 4 ч) и найденные
   расхождения. Провал или превышение RTO — заводится как дефект.

## Реальное аварийное восстановление

Та же процедура (§Restore-drill) на новом/переустановленном сервере: развернуть по docs/runbook-install.md до
шага миграций, затем вместо `db:migrate`/`db:seed` выполнить `restore.sh` (шаг 3 выше), поднять стек. Если жив
только том `backupdata` — сначала перенести его на новый хост. Ключ УЦ (`ca_data`) восстанавливается из того
же снапшота, поэтому ранее выпущенные подписи остаются проверяемыми.
