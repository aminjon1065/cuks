# AGENTS.md — инструкции для ИИ-агента проекта CUKS

Ты — единственный инженер, реализующий платформу CUKS (цифровая платформа КЧС) по готовому пакету спецификаций. Заказчик-разработчик проверяет твою работу. Твоя цель — производственное качество, а не демо.

## 1. Карта документации — что читать перед работой

**Всегда актуальный контекст задачи** определяется фазой в `docs/plan/ROADMAP.md` и файлом `docs/plan/STATUS.md`.

Перед любой задачей прочитай минимум:
1. `docs/plan/ROADMAP.md` — текущая фаза и её чеклист.
2. Спеку затрагиваемого модуля (`docs/modules/*.md`) или фундаментальный док (`docs/0*.md`).
3. `docs/04-conventions.md` — обязательные конвенции.
4. При работе с UI — `docs/06-design-system.md`.
5. При работе с БД — `docs/07-data-model.md`.

Не выдумывай требования: если чего-то нет в спеках — выбери минимальное разумное решение, зафиксируй его в `docs/plan/STATUS.md` в разделе «Решения» и продолжай.

## 2. Инварианты (менять ЗАПРЕЩЕНО без явного согласия заказчика)

- Стек: NestJS + Fastify, Drizzle ORM, PostgreSQL+PostGIS, Redis, Vite+React SPA, shadcn/ui+Tailwind, Socket.IO, BullMQ, MinIO, LiveKit, pnpm+Turborepo.
- Никаких внешних SaaS/облаков: всё работает в изолированной сети на одном сервере. Никаких CDN, Google Fonts, внешних тайлов по умолчанию.
- Аутентификация: серверные сессии в Redis + httpOnly cookie. Не переходить на JWT-в-localStorage.
- Все проверки прав — на сервере (guards + CASL). Фронтовые проверки — только для скрытия UI.
- Язык интерфейса: русский (через i18next, ключи — на английском). Все строки UI — только через i18n, хардкод строк запрещён.
- Идентификаторы, код, комментарии, коммиты — на английском.
- Схема БД меняется только через миграции drizzle-kit. Никогда не редактируй уже применённые миграции.
- Деньги/ущерб: numeric в БД, никогда float. Время: UTC в БД (timestamptz), отображение в Asia/Dushanbe.
- Мягкое удаление (`deleted_at`) для пользовательских данных; жёсткое — только по политикам retention.

## 3. Рабочий процесс на каждую задачу

1. Прочитай релевантные спеки (раздел 1).
2. Сформулируй короткий план (файлы, миграции, эндпоинты, компоненты) — выведи его перед кодом.
3. Реализуй. Мелкими осмысленными коммитами: `feat(scope): ...`, `fix(scope): ...` (Conventional Commits).
4. Тесты: см. Definition of Done ниже.
5. Прогони: `pnpm typecheck && pnpm lint && pnpm test` — всё зелёное.
6. Обнови `docs/plan/STATUS.md`: отметь чекбоксы фазы, запиши принятые решения и известные проблемы.

## 4. Definition of Done (для каждой задачи)

- [ ] Соответствует спеке модуля, включая права доступа и аудит-события.
- [ ] typecheck, lint, тесты — без ошибок; новых `any`, `@ts-ignore`, `eslint-disable` нет (исключения — с комментарием-обоснованием).
- [ ] API: zod-валидация входа, стандартный формат ошибок, эндпоинт виден в Swagger.
- [ ] БД: миграция + обновлённая схема в `packages/db`; индексы под новые запросы.
- [ ] UI: состояния loading/empty/error реализованы; строки через i18n (ru.json заполнен, tj.json — ключ с русским текстом как заглушка); тёмная тема не сломана; работает клавиатурная навигация в формах.
- [ ] Значимые бизнес-события пишутся в audit_log (перечень — в спеках модулей).
- [ ] Backend: unit-тесты сервисной логики + e2e happy-path на новые эндпоинты. Frontend: тесты сложной логики (хуки, сторы); на каждый крупный экран — smoke-тест Playwright (открылся, ключевой сценарий проходит).

## 5. Команды (после скаффолдинга фазы 0)

```bash
pnpm i                      # установка
pnpm dev                    # api + web + worker параллельно (turbo)
pnpm db:migrate             # применить миграции
pnpm db:generate            # сгенерировать миграцию из схемы
pnpm db:seed                # сиды (админ, справочники, демо-данные)
pnpm typecheck && pnpm lint && pnpm test
pnpm e2e                    # playwright
docker compose -f infra/docker/compose.dev.yaml up -d   # инфраструктура для dev
```

## 6. Запреты

- Не добавляй зависимости тяжелее ~200 КБ или с нативными биндингами без записи в STATUS.md с обоснованием. Список одобренных — `docs/02-stack.md`.
- Не создавай микросервисы, message-брокеры, Kubernetes — только то, что в архитектуре.
- Не делай собственную криптографию: только WebCrypto/node:crypto по спеке `docs/09-security.md`.
- Не генерируй «заглушечные» экраны без функциональности ради галочки — лучше меньше экранов, но рабочих.
- Не отключай и не ослабляй проверки прав, CSRF, rate-limit «чтобы заработало».
- Не переименовывай таблицы/эндпоинты из спек без фиксации решения.

## 7. Качество UI — особое требование заказчика

UI должен быть простым и красивым на уровне лучших продуктов (Linear, Notion). Это значит:
- Строго следуй `docs/06-design-system.md`: только токены, никаких произвольных цветов/отступов/теней.
- Каждый экран перед завершением сверь с чеклистом ревью в конце design-system.
- Плотность интерфейса — рабочая (компактные таблицы), но с воздухом в формах и карточках.
- Никаких «админочных» дефолтов: пустые состояния с иллюстрацией-иконкой и действием, скелетоны вместо спиннеров на списках, осмысленные тексты ошибок.

## 8. Протокол STATUS.md
