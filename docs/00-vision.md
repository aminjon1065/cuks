# 00. Видение продукта

## Что строим

**CUKS** — единая цифровая платформа Комитета по чрезвычайным ситуациям (КЧС): рабочее место каждого сотрудника от оперативного дежурного до председателя. Заменяет зоопарк из бумаги, Word-файлов, мессенджеров и разрозненных карт одной системой на собственном сервере Комитета.

Шесть модулей на общем фундаменте (пользователи, орг-структура, права, файлы, уведомления, аудит, поиск):

| # | Модуль | Суть | Спека |
|---|--------|------|-------|
| 1 | **ГИС и аналитика** (основной) | Реестр ЧС, карты в вебе + интеграция с QGIS/ArcGIS, дашборды, отчёты | modules/10 |
| 2 | Документооборот | Полный корпоративный ДОУ: журналы, маршруты, резолюции, контроль, ЭЦП | modules/11 |
| 3 | Файловый менеджер | Личные и общие файлы, версии, шаринг, предпросмотр | modules/12 |
| 4 | Чат | Каналы, личные сообщения, упоминания, файлы | modules/13 |
| 5 | Звонки и конференции | Аудио/видео, демонстрация экрана, запись | modules/14 |
| 6 | Задачи | Доски Trello/Jira-lite, «Мои задачи», связи с ЧС и документами | modules/15 |

## Контекст заказчика

- КЧС — государственный орган: центральный аппарат + областные управления + городские/районные отделы. Иерархия жёсткая, работа документоцентричная, дисциплина сроков.
- Профиль рисков региона: землетрясения, сели, лавины, оползни, паводки/наводнения, ливни и снегопады, засухи, техногенные аварии.
- Инфраструктура: **один сервер + интернет**. Возможны каналы низкого качества у районных отделов → приложение должно быть лёгким, а карты — self-hosted.
- Требования госоргана: данные не покидают сервер, полный аудит действий, разграничение доступа вплоть до грифа «ДСП», русский интерфейс (таджикский — вторым языком).
- Разработка: один разработчик + ИИ-агент. Значит: максимально простая эксплуатация, минимум движущихся частей, скучные проверенные технологии.

## Пользователи (персоны)

| Персона | Что делает в системе | Критично |
|---|---|---|
| **Оперативный дежурный** (24/7, ситуационный зал) | Вносит донесения о ЧС, ставит точку на карте, оповещает, создаёт поручения | Скорость ввода (< 2 мин на донесение), тёмная тема, крупные статусы |
| **Руководитель** (председатель, начальники управлений) | Смотрит сводку/дашборд, накладывает резолюции, подписывает, проводит совещания | Всё в 1–2 клика, мобильный просмотр, «на подпись/на контроле» |
| **Делопроизводитель / канцелярия** | Регистрирует входящие/исходящие, ведёт журналы, контроль сроков | Быстрые формы, нумерация без ошибок, печать карточек |
| **Аналитик ГИС** | Ведёт слои, импортирует данные, строит карты и отчёты, работает из QGIS | Прямой доступ к PostGIS, WMS/WFS, импорт/экспорт форматов |
| **Сотрудник** | Задачи, документы на исполнение, чат, звонки, файлы | Единый inbox: всё, что требует его действия |
| **Администратор платформы** | Пользователи, права, справочники, бэкапы | Понятная админка, аудит |

## Сквозной сценарий (нить, связывающая все модули)

1. **Донесение**: дежурному звонят из района — сель перекрыл дорогу. Он создаёт запись в реестре ЧС: вид «сель», точка на карте, предварительные данные о пострадавших. *(ГИС)*
2. **Оповещение**: система шлёт уведомления причастным по матрице оповещения; дежурный одним действием создаёт канал ЧС в чате. *(Чат, уведомления)*
3. **Штаб**: начальник управления запускает видеоконференцию из этого канала, включает запись. *(Звонки)*
4. **Поручения**: по итогам создаются задачи с исполнителями и сроками, привязанные к карточке ЧС. *(Задачи)*
5. **Документы**: готовится приказ о выделении сил — маршрут согласования, подпись председателя ЭЦП, регистрация в журнале. Приказ связан с ЧС. *(Документооборот)*
6. **Материалы**: фото с места, акты обследования — в файлах карточки ЧС, доступны по правам. *(Файлы)*
7. **Аналитика**: через месяц аналитик строит отчёт «ЧС за период» с картой и динамикой, выгружает XLSX для доклада, а зоны риска редактирует в QGIS прямо в PostGIS. *(Аналитика/ГИС)*

Любая сущность (ЧС, документ, задача, файл, канал, запись встречи) может быть связана с любой другой — универсальные связи `entity_links` (см. 07-data-model).

## Принципы продукта

1. **Простота важнее функциональности.** Каждая фича проходит тест: «поймёт ли делопроизводитель 50+ лет без обучения?». Меньше настроек, больше разумных умолчаний.
2. **Один inbox.** Всё, что требует действия пользователя (согласовать, подписать, исполнить, задача, упоминание), сходится в единую ленту «Требует внимания» на дашборде.
3. **Self-hosted без компромиссов.** Отключили внешний интернет — платформа работает полностью (карты, шрифты, всё локально).
4. **Скорость на слабых каналах.** Первая загрузка ≤ 500 КБ критического JS, code-splitting по модулям, векторные тайлы.
5. **Аудит всего.** Госорган: кто, что, когда — восстанавливается за минуты.
6. **Красиво = доверие.** Уровень UI — как у лучших коммерческих продуктов. Это официальное требование, см. 06-design-system.

## Критерии успеха v1

- Дежурная смена ведёт 100% донесений о ЧС в системе; сводка за сутки формируется автоматически.
- Канцелярия регистрирует все входящие/исходящие; контроль сроков — из системы, а не из тетради.
- Руководство подписывает внутренние документы ЭЦП без бумаги.
- Аналитик подключается к слоям из QGIS и ArcGIS без помощи разработчика (по инструкции).
- 200–500 активных пользователей на одном сервере без деградации.

## За рамками v1 (осознанно)

- Нативные мобильные приложения (веб адаптивен для чтения/чата; PWA — фаза 7+).
- Прогнозные ML-модели, автоматический сбор данных с датчиков/гидропостов (заложены таблицы, интеграции позже).
- Интеграция с государственной квалифицированной ЭЦП (этап 2, после внутренней ЭЦП).
- Редактирование office-документов в браузере (рассмотрим ONLYOFFICE позже; v1 — версии + скачивание).
- Межведомственный обмен (СЭД-шлюзы).
